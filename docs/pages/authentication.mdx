Authentication and permissions solve two different problems:

- **Authentication** confirms who the current user is (`userId`) and optionally
  which workspace they belong to (`organizationId`).
- **Permissions** define what that authenticated user can do in Liveblocks
  resources such as rooms, comments, and feeds.

## How permissions work

A user has access to a room through a list of permissions.

A permission has a resource (e.g. `room`, or `room:comments`) and a scope
(`read`, `write`, or `none`).

### Default scopes

By setting the permission for the resource `room`, you define the default scope
for all the resources of the room. You can choose one of these three scopes:

- `room:read` => by default, the user will have read access to everything in the
  room.
- `room:write` => by default, the user will have write access to everything in
  the room.
- `room:none` => by default, the user will have no access to the room.

### More granular permissions

You can opt in to or opt out of access to specific room resources:

- **Presence** with `room:presence:read|none`.
- **Storage** with `room:storage:write|read|none`.
- **Comments** with `comments:write|read|none`.
- **Feeds** with `room:feeds:write|read|none`.

**Here are some examples**

1. Write access to everything except read-only access to storage:

```
[
  "room:write",  // default
  "room:storage:read"  // opt-out storage
]
```

2. Only access to Presence and Comments:

```
[
  "room:none",  // default
  "room:presence:read",  // opt-in presence
  "room:comments:write"  // opt-in comments
]
```

## Authentication systems

Choose one of these approaches for generating auth tokens:

- [ID token authentication](/docs/authentication/id-token) (recommended): define
  permissions on rooms with Liveblocks APIs, then authenticate users with
  `identifyUser`.
- [Access token authentication](/docs/authentication/access-token): define
  allowed room IDs (often with wildcard patterns) every time you issue a token.

<Banner title="Use ID tokens for most production apps" type="warning">

ID tokens scale better for complex permission models (organization, group, and
user-level access). Access tokens are useful when you want to keep permission
logic entirely in your own auth endpoint.

</Banner>
